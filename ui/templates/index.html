Configured Thresholds Set via env vars on the bitagent core. Restart required to change.
KnobEnv varDefaultWhat it does
enabled BITAGENT_EVIDENCE__LIVENESS__ENABLED true Master toggle. When false, no observations are emitted and Torznab responses are unfiltered (zero overhead).
stall_threshold_hours …__STALL_THRESHOLD_HOURS 4 How long a torrent must stay in stalledDL/metaDL before it's eligible to be marked dead.
min_observations …__MIN_OBSERVATIONS 2 How many independent suspect observations are required. Guards against transient network issues.
blacklist_ttl_days …__BLACKLIST_TTL_DAYS 30 How long until a dead torrent gets re-checked. Lower = faster recovery if it comes back; higher = less DHT load.
revalidate_interval …__REVALIDATE_INTERVAL 6h How often the revalidation worker wakes up to re-check a batch of dead torrents.
revalidate_min_peers …__REVALIDATE_MIN_PEERS 2 Minimum DHT peers that must respond to a get_peers query for the torrent to be marked alive again.
exclude_private_tracker_grabs …__EXCLUDE_PRIVATE_TRACKER_GRABS true When a *arr imports a torrent in the qB private category, don't credit it as alive evidence — the file came from a private tracker, not BitAgent's catalog.
How it works

Inputs. Every /api/v2/torrents/info poll cycle, qBittorrent reports the state of each torrent it knows about. seeding/uploading/downloading = alive. stalledDL/metaDL/error/missingFiles = suspect. Plus every successful *arr import webhook = alive.

Resolution. A torrent is marked dead only after min_observations independent suspect observations across at least stall_threshold_hours. Any single alive observation immediately wins.

Output. Dead infohashes are filtered out of Torznab responses before they reach Sonarr/Radarr — no stale grabs.

Revalidation. Every revalidate_interval, the worker picks a batch of dead torrents and fires a DHT get_peers query. If revalidate_min_peers respond, the torrent is reinstated as alive. Otherwise it stays dead until the next sweep.
